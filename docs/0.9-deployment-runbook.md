# AutoScript Hub 0.9 局域网部署与运维

本文适用于 `linux/arm64`（aarch64）与 `linux/amd64`（x86-64）单机部署。0.9 使用 SQLite，只允许一个服务端副本。

## 首次部署

```bash
unzip autoscript-hub-server-deploy-0.9.0.zip
cd autoscript-hub-server
cp deploy/.env.example deploy/.env
id -u
id -g
```

编辑 `deploy/.env`：

- 把 `JWT_SECRET` 与 `ADMIN_PASSWORD` 替换为真实随机值。
- `AUTOSCRIPT_DATA_DIR` 使用主机绝对路径，例如 `/opt/autoscript-hub/data`。
- `AUTOSCRIPT_UID`、`AUTOSCRIPT_GID` 填入运行目录所有者的 UID/GID。
- 默认绑定 `0.0.0.0`，因此局域网设备可以访问；不要直接暴露到公网。

```bash
sudo install -d -o "$(id -u)" -g "$(id -g)" /opt/autoscript-hub/data
docker compose --env-file deploy/.env -f deploy/compose.yaml up -d
docker compose --env-file deploy/.env -f deploy/compose.yaml ps
curl --fail http://127.0.0.1:8000/api/health/ready
```

局域网地址为 `http://服务器IP:AUTOSCRIPT_PORT`。就绪响应中的 `database`、`data_dir`、`migration` 应全部为 `ok`。

## 国内网络和镜像代理

下面的源码构建命令只适用于完整源码仓库；独立部署 ZIP 使用已经发布的 GHCR 镜像，不包含应用源码。完整源码仓库默认使用 Docker Hub。若 Docker Hub 不可达，可在 `deploy/.env` 设置兼容代理，不必修改 Docker daemon：

```dotenv
AUTOSCRIPT_BASE_REGISTRY=docker.m.daocloud.io/library
```

然后构建：

```bash
docker compose --env-file deploy/.env \
  -f deploy/compose.yaml -f deploy/compose.local.yaml up -d --build
```

若 `docker info` 显示不可达的 HTTP/HTTPS Proxy，应先恢复该代理，或由主机管理员修改 Docker 服务代理并评估重启对现有容器的影响。

## 日常检查和日志

```bash
curl --fail http://127.0.0.1:8000/api/health
curl --fail http://127.0.0.1:8000/api/health/ready
docker compose --env-file deploy/.env -f deploy/compose.yaml ps
docker compose --env-file deploy/.env -f deploy/compose.yaml logs --tail=200 server
docker compose --env-file deploy/.env -f deploy/compose.yaml logs -f server
```

上面的完整命令等价于在已设置默认 Compose 文件时使用 `docker compose logs server`；运维时应保留 `--env-file` 和 `-f` 参数以避免读错项目。

容器日志写入 stdout/stderr；数据库、脚本、业务日志、备份和临时文件均位于 `AUTOSCRIPT_DATA_DIR` 映射的 `/data`。

## 在线备份

备份通过容器内 Python 和 SQLite 在线备份 API 执行，主机不需要额外安装 Python 包：

```bash
AUTOSCRIPT_DATA_DIR=/opt/autoscript-hub/data sh ops/server/backup.sh
```

同一主机运行多套 Compose 时，为所有运维命令增加与启动时一致的 `AUTOSCRIPT_PROJECT_NAME`。

命令返回 `/data/backups/<时间>-<版本>`。目录包含数据库副本、脚本、清单和 SHA-256。真实 `deploy/.env` 不包含在数据备份中，应另行保存。

## 停止恢复

恢复会覆盖当前数据，必须先停止服务，并指定 `/data/backups/...` 或对应的主机绝对路径：

```bash
docker compose --env-file deploy/.env -f deploy/compose.yaml stop server
AUTOSCRIPT_DATA_DIR=/opt/autoscript-hub/data \
  sh ops/server/restore.sh /data/backups/<时间>-<版本>
docker compose --env-file deploy/.env -f deploy/compose.yaml up -d server
curl --fail http://127.0.0.1:8000/api/health/ready
```

`restore.sh` 会先验证清单、大小和 SHA-256，校验失败不会恢复。

## 升级

升级脚本先备份，再拉取目标镜像、重建容器并检查就绪；失败会调用回滚：

```bash
AUTOSCRIPT_DATA_DIR=/opt/autoscript-hub/data \
  AUTOSCRIPT_SERVER_IMAGE=ghcr.io/czf39631/autoscript-hub-server:0.9.0 \
  sh ops/server/upgrade.sh 0.9.1
```

升级成功后，把 `deploy/.env` 中的 `AUTOSCRIPT_SERVER_IMAGE` 固化为完整 SemVer 镜像引用，防止下次手工 Compose 启动回到旧标签。升级失败时脚本会记录并使用升级前容器的不可变镜像 ID 执行恢复。

完全离线的局域网主机可以由管理员先用 `docker load` 导入目标镜像，然后显式设置 `AUTOSCRIPT_SKIP_PULL=1`：

```bash
AUTOSCRIPT_DATA_DIR=/opt/autoscript-hub/data \
  AUTOSCRIPT_SERVER_IMAGE=ghcr.io/czf39631/autoscript-hub-server:0.9.0 \
  AUTOSCRIPT_SKIP_PULL=1 \
  sh ops/server/upgrade.sh 0.9.1
```

该开关只跳过远端拉取；脚本仍会备份、重建、迁移和检查就绪。默认不设置时始终拉取发布镜像。

## 人工回滚

必须同时指定旧镜像版本和与它匹配的升级前备份：

```bash
AUTOSCRIPT_DATA_DIR=/opt/autoscript-hub/data \
  sh ops/server/rollback.sh 0.9.0 /data/backups/<升级前备份>
```

回滚完成后再次检查 `/api/health/ready`、登录、脚本列表和日志。

## 持久化验收

1. 登录并上传一个测试脚本，执行一次并产生日志。
2. 执行 `docker compose ... down`，不得删除 `AUTOSCRIPT_DATA_DIR`。
3. 再次 `docker compose ... up -d`。
4. 确认账号、脚本、运行记录和日志仍存在。
5. 做一次备份、产生新数据、停止服务、恢复备份，确认新数据消失而备份时的数据存在。
